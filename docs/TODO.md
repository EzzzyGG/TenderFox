# Стратегический план развития (12 месяцев)

## Q1 — MVP и первые продажи (через таргет/рекламу)

### 1) Цель Q1
Сделать работающий агрегатор тендеров: поиск → фильтр → подписка (на сайте) → уведомления (Telegram) + сайт-лендинг.

### 2) MVP (обязательное)
- [x] Источник данных: рабочий REST-источник (через GosPlan v2 test как обёртку над ЕИС/zakupki)
- [x] Нормализация результата (единый JSON формат)
- [x] БД Postgres: таблицы tenders / subscriptions / deliveries
- [x] Alembic миграции
- [x] API:
  - [x] `GET /search?keyword=&region=` — реальные результаты
  - [x] `GET /tenders/{id}` — детали (кеш в БД)
  - [x] `POST /subscriptions` — создать подписку (старое: chat_id)
  - [x] `GET /subscriptions` — список подписок

- [ ] Пользователи и авторизация (PHONE-first)
  - [ ] Нормализация телефонов СНГ в E.164 (минимум коды: +7, +375, +380, +374, +994, +995, +996, +998, +992, +993, +373)
  - [ ] Гибридная верификация телефона:
    - [ ] попытка через Telegram contact (если возможно)
    - [ ] fallback через SMS-код (на MVP можно stub)
  - [ ] JWT авторизация для сайта (Next.js)
  - [ ] Привязка Telegram к `user_id` (таблица `telegram_links`)

- [ ] Подписки на сайте
  - [ ] Перевести `subscriptions` на `user_id` (в API убрать зависимость от chat_id)
  - [ ] Ограничить CRUD подписок авторизованным пользователем

- [ ] Telegram (как канал доставки)
  - [ ] onboarding: привязка к user
  - [ ] отправка уведомлений по подписке (через scheduler)

- [ ] Планировщик (end-to-end)
  - [ ] периодическая проверка подписок и доставка в Telegram
  - [x] защита от дублей (dedupe по tender_id + subscription_id) — заложено deliveries

- [x] Запуск:
  - [x] Dockerfile + docker-compose (api + postgres + redis)
  - [x] `.env.example`

- [x] CI:
  - [x] GitHub Actions: lint + tests
